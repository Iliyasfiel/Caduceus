## 一、产品概述

### 1.1 产品定位

Caduceus 是一款**轻量级、开源的协同工作平台**。它不是一个传统的 OA 或项目管理软件，而是一个**聚焦角色间信息流转的信息枢纽**——让多人多角色的协作团队能够以透明的信息共享和灵活的流程编排，高效地协同完成工作。

### 1.2 产品核心理念

- **流程是协作的加速器，而非限制协作的锁链**
- **信息全透明，操作有边界** — 所有协作者可查看任务的全部信息，但谁能执行什么操作由角色和权限控制
- **管理员可在线自定义流程** — 通过可视化管线编辑器，无需二次开发即可适应不同协作场景

### 1.3 产品目标

- 帮助小团队（50 人以内）高效管理多角色分工协作
- 降低流程变更的维护成本，让管理员而非开发人员来配置流程
- 作为开源项目（Apache 2.0），让社区能够自由使用、定制和贡献

## 二、目标用户与场景

### 2.1 目标用户

| 用户角色 | 描述 | 核心需求 |
|---------|------|---------|
| **发起人** | 提出工作需求的员工 | 快速创建任务、补充信息、查看进展 |
| **执行人** | 具体执行工作的员工 | 清晰了解自己的任务、反馈进度、获取所需信息 |
| **主管** | 业务小组的负责人 | 管理本组任务分配、掌握进度、处理超时 |
| **管理员** | 系统运维和配置人员 | 配置流程、管理用户和角色、维护系统 |

### 2.2 典型使用场景

**场景一：行政接待**

公司行政部门需要接待来访的重要客户。一次接待任务涉及多个角色并发工作：

- 发起人（行政专员）创建接待任务，填写客户信息、时间、人数（初始信息可能不完整）
- 讲解员准备讲解内容，查看参观安排
- 接待员协调用餐和休息安排
- 司机确认派车时间、接送地点
- 中途发生行程变更（如客户临时增加人数），所有角色实时获取更新
- 主管可查看整体进度，协调资源

**场景二：会议活动**

从方案策划到现场执行，涉及策划人、场地、物料、嘉宾联络等多个角色：

- 策划人创建任务，分解场地确认、嘉宾邀请、物料准备等子任务
- 子任务按顺序执行（先确认场地，再通知嘉宾）
- 各环节信息在同一任务下透明共享

**场景三：其他可复用的场景**

Caduceus 不限定于特定行业。任何具有以下特征的工作场景都可适用：

- 多人多角色参与同一件事
- 信息需要共享而非隔离
- 流程需要灵活调整而非固定
- 涉及可复用的资源或模版

### 3.3 管线编辑器

**设计理念：管线定义的是信息流，而非工作流。** 管线用于结构化展示"任务包含哪些环节、各环节谁参与、需要什么信息"，但不阻塞信息修改。所有角色对任务信息的修改实时同步给所有参与者。

| ID | 需求 | 优先级 |
|----|------|--------|
| P-01 | 管理员可进入可视化编辑器创建/修改管线 | P0 |
| P-02 | 编辑器提供拖拽式节点编排界面，画布上只有"任务节点"一种视觉元素 | P0 |
| P-03 | 每个任务节点可配置：该阶段的字段定义、参与的角色、关联的资源类型 | P0 |
| P-04 | 节点之间可连线，表达阶段推进方向 | P0 |

| P-06 | 字段可标记"优先展示角色"，该角色的使用者在该字段置顶看到 | P0 |
| P-07 | 支持公共字段（不关联任何角色，对所有参与者平等展示） | P0 |
| P-08 | 当一个阶段的工作基本完成时，参与者可手动标记该阶段完成 | P1 |
| P-09 | 标记阶段完成不阻塞信息修改，已完成节点的信息仍可被修改并同步 | P1 |

## 六、用户界面与交互要点

### 6.1 页面结构

- **导航**：左侧边栏，展示 Caduceus 品牌、功能菜单（仪表盘、任务、资源库、管线配置等）
- **头部**：通知铃铛（未读计数）、用户信息、退出登录
- **主体**：内容区域，展示具体页面

### 6.2 体验要点

- 任务详情页采用"共用结构 + 字段按角色优先级排序"——所有角色使用同页面，但对自己重要的信息置顶显示
- 任务列表页提供"只看与我相关"的过滤切换
- 管线编辑器采用 Vue Flow，左侧节点类型面板 + 中间画布区域 + 右侧配置面板
- 所有操作应有即时反馈（状态变更后页面自动更新，无需手动刷新）
- 表单单据流程支持草稿暂存，后续可继续编辑

## 八、术语表

| 术语 | 定义 |
|------|------|
| **管线** | 由管理员创建的可视化"信息流"图，定义了任务各阶段的角色、字段和资源配置关系，是结构化指引而非执行锁链 |
| **节点** | 管线中的任务阶段，内部配置了该阶段涉及的字段定义、参与角色和资源类型 |
| **任务池** | 未指定执行人的任务集合，拥有对应角色的执行人可自行接取 |
| **资源库** | 可复用的信息库，管理员可自定义资源类型和字段 |
| **角色组** | 具有相同业务角色的用户集合，如"讲解组"、"车队" |
| **主管** | 角色组的管理者，可管理本组成员的任务分配 |
| **优先级角色** | 任务字段的一个标记，表示该字段对哪些角色最为重要 |
| **公共字段** | 不关联特定角色的字段，对所有参与者平等展示 |

## 九、附录

### 9.1 术语对比：Caduceus vs 传统OA / 自动化工作流

| 维度 | 传统 OA / n8n 类 | Caduceus |
|------|---------|---------|
| 信息策略 | 信息隔离，按角色设置查看权限 | 信息全透明，所有协作者可查看全部信息 |
| 流程目的 | 管控和审批，控制流转 | **信息流**赋能协作，高效传递 |
| 流程约束 | 硬约束，上一节点未完成则阻塞下一节点 | 软约束，阶段完成仅作为参考提示，不阻塞信息修改 |
| 流程调整 | 需要二次开发或复杂配置 | 管理员拖拽编排，立即可用 |
| 字段定义 | 节点独占字段，数据随节点流转 | 字段全局共享，一处修改即时同步 |

### 9.2 与老系统（ASP+Access）的对比

| 维度 | 老系统（IDEAOPEN） | Caduceus |
|------|-------------------|---------|
| 技术栈 | ASP + VBScript + Access | Django + Vue 3 + PostgreSQL |
| 数据管理 | 15个碎片化Access数据库 | 统一 PostgreSQL，JSONB 灵活字段 |
| 流程配置 | 硬编码在 ASP 文件中 | 可视化管线编辑器 |
| 角色管理 | 数字 p_id 硬编码 | 灵活的角色-权限体系 |
| 实时性 | 无实时推送（3分钟自动刷新） | WebSocket 实时推送 |
| 扩展性 | 新功能需修改代码 | 可自定义角色/资源/管线 |
| 部署 | IIS + Windows | Docker 一键部署，跨平台 |
