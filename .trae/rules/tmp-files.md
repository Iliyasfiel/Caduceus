# 临时文件与演示数据（虚拟环境 / demo-data / dev-tmp）

> 集中管理三类"非正式代码"的存放策略。所有规则条目**直接以本文件为权威**——如与 `project_rules.md` 总览冲突，以本文为准。

## 1. 虚拟环境

- **禁止**将任何 Python 虚拟环境提交到远程仓库，包括但不限于：`venv/`、`.venv/`、`env/`、`.env-virtual/`、`virtualenv/` 等。
- 依赖通过 `requirements.txt`（或 `pyproject.toml`）管理。Clone 仓库后应在本地重建虚拟环境：
  ```bash
  python3 -m venv backend/.venv
  source backend/.venv/bin/activate
  pip install -r backend/requirements.txt
  ```
- 已在 `.gitignore` 中忽略 `venv/` 与 `.venv/`，避免误提交。

### 部署方式分两个阶段（2026-07-06 补充）
- **日常开发**：使用简单虚拟环境（`python -m venv` + `pip install -r requirements.txt`），不引入 Docker 复杂度。
- **即将上线 / 上线前测试**：可以使用 `docker compose up` 等容器化方式验证生产一致性与依赖收敛，但**仅用于上线前验证**，不作为日常开发流程。

## 2. 演示数据（demo-data）

- 根目录下的 `demo-data/` 目录用于存放**开发演示用的临时数据**（如示例资源类型、示例任务等），用于手动验证页面效果。
- **`demo-data/` 允许提交**到远程仓库（与 `node_modules/`、`.venv/` 等被忽略的目录形成对比）。

### 存放策略（2026-07-06 明确）
- 提交 `demo-data/*.json` 等数据文件本体到云端；
- 部署后由**后端服务或种子脚本自动读取并导入**为预设的测试 / 展示数据；
- 默认即"开箱即用"——新部署完即可看到演示数据，无需手动操作。

### 存放内容规范
- ✅ 允许：非敏感的、可复现的、用于手动验证的演示数据（如示例资源类型 JSON、示例用户、示例任务）。
- ❌ 禁止：生产数据、真实用户隐私、任何含 API Key/Token/密码的内容。

### 生产环境注意
若同一份 `demo-data/` 既用于开发又用于部署，需在生产入口（如 `manage.py` 或 `docker-compose.yml`）通过环境变量（如 `SEED_DEMO_DATA=false`）控制是否导入，避免污染生产数据。

详见 `demo-data/README.md` 与 `README.md` 的"演示数据（demo-data）"章节。

## 3. 临时文件（dev-tmp）

开发过程中产生的**临时文件**（如对话中临时整理出的需求讨论文档、生成图片用的脚本文件、一次性草稿、调试用的样例数据等），必须放在**项目根目录下的 `dev-tmp/`** 文件夹中集中管理，**不得散落在仓库其他位置**。

### 适用范围
- ✅ 必须放入：`dev-tmp/` — 临时需求讨论文档、图片生成脚本、一次性草稿、调试样例。
- ❌ 禁止散落：根目录、`.trae/`、`designingDocument/`、`demo-data/`、`backend/`、`frontend/` 等正式目录（避免污染正式产物）。

### 适用边界 — `dev-tmp` vs `demo-data` vs 设计文档
- **`dev-tmp/`**：还在**讨论中、尚未定型**的临时草稿（如"今天尝试的某方案草图"、"会话中临时列出的待确认点"），用于个人 / 跨设备同步。
- **`designingDocument/`**：已经定型、提炼过的**正式设计文档**（如 `caduceus-prd.md`、`caduceus-design.md`），可作为开发依据被引用。
- **`demo-data/`**：用于**演示 / 测试 / 展示**的可复现数据（如示例任务 JSON），与运行时数据生成相关。

### 提交策略（2026-07-06 明确）：按价值分级提交，并非全量纳入跟踪。

**应该提交**：
- 设计草稿（如 `dev-tmp/design/component-api-draft.md`）— 跨设备同步，避免单点丢失；
- 需求讨论结论（如会议纪要、待办项提炼）— 多人 / 跨设备可参考；
- 一次性的工程操作脚本（如 `dev-tmp/git/sync-to-cloud.sh`）— 复用价值高；
- 已提炼但尚未合并进正式文档的**纯文档类临时产物**。

**不应提交**（保持本地 / 未追踪）：
- 图片生成脚本的一次性输出（生成的 PNG/JPG/SVG 等大文件）；
- 调试样例（一次性的 curl 脚本响应、临时 SQL 输出）；
- 私人笔记、临时 TODO、调试打印输出；
- 任何含 API Key、Token、个人隐私的临时文件；
- 体积过大的纯产物（如生成的视频、二进制中间产物）。

**判定标准：问自己"半年后重新打开这个项目时，这个文件是否还有用？"**
- 是 → 提交
- 否 → 不提交（保留本地即可，或干脆删除）

### 当前 git 跟踪策略
`dev-tmp/` 纳入 git 跟踪（`.gitignore` 不再忽略 `dev-tmp/`，于 2026-07-06 调整），但**只跟踪上述"应该提交"的子集**；其余一次性 / 隐私 / 大体积文件应保持本地不追踪或放 `.gitignore` 例外规则。

### 提交前清理提示（IMPORTANT）
每次准备把 `dev-tmp/` 下的文件提交进 git 之前，**必须**先停下，向用户列出**待提交文件清单 + 内容概要**，询问是否有要删除的或同步沉淀到正式文档的；获得用户明确答复后再执行 `git add` + `commit`。

- 适用场景：`git add dev-tmp/` / `git commit` 涉及 `dev-tmp/` 下任何文件 / 子目录时。
- 不适用：`dev-tmp/` 下只有细微 typo 修正、纯格式调整等显然无沉淀价值的小改动——但即便如此，**至少要在 commit message 里说明**。

### 删除确认流程（IMPORTANT）
清理 `dev-tmp/` 内任何文件 / 子目录前，必须**先向用户确认**该临时内容是否需要同步沉淀到项目正式文档（如 `designingDocument/`、`README.md`、`docs/` 等）：
1. 列出待删除的文件清单及其内容概要；
2. 询问用户："是否需要将上述内容同步到项目正式文档？（如不需要再删除）"；
3. 用户明确答复"无需同步"或"已同步"后，方可执行删除。

### 同步建议场景
- 需求讨论文档中提炼出的最终结论 → 合并进 `designingDocument/caduceus-prd.md` 或对应 design 文档；
- 图片生成脚本中可复用的逻辑 → 沉淀为正式工具或文档示例；
- 调试样例中有保留价值的部分 → 转存到 `demo-data/`。

### 统一存放点
项目**只有**根目录下 **一个** `dev-tmp/` 目录作为临时文件总入口；**禁止**在 `designingDocument/`、`backend/`、`frontend/` 等子目录下再创建 `dev-tmp/`（2026-07-06 之前曾在 `designingDocument/dev-tmp/` 存放 UI 临时文档，已迁移到根目录并删除子目录副本）。

### 分类子目录约定
根据临时内容的性质，可在 `dev-tmp/` 下创建分类子目录，但子目录内文件仍须遵守本节所有清理规则。当前已建立的分类：
- `dev-tmp/design/` — 设计文档相关临时草稿（组件 API / 设计原则 / Polish 记录 / 路线图等）
- `dev-tmp/git/` — 工程操作相关临时文件（推送脚本、迁移脚本等一次性指南）

### 每个子目录的索引
建议每个分类子目录放一个 `README.md` 列出内容清单与清理规则，便于后期清理时一眼看清全貌。
